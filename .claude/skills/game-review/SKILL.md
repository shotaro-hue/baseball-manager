# Game Review スキル

SPEC.md・ROADMAP.md・実際のコードを照合し、ゲームの理想形との乖離を評価して優先度付きのアクションリストを生成するワークフロー。

## ワークフロー

Make a todo list for all the tasks in this workflow and work on them one after another.

### 1. SPEC.md を読み込む

`SPEC.md` を読み込み、以下を把握する:
- ゲームの設計目標・コンセプト
- 全機能の一覧と詳細仕様
- 数値バランスの設計意図
- ユーザー体験・ゲームプレイフローの理想像

### 2. ROADMAP.md を読み込む

`ROADMAP.md` を読み込み、各 Tier の実装状況を整理する:
- ✅ 完了済みの機能
- 🔶 部分実装の機能（何が足りないか）
- 🔴 未着手の機能
- 既知のバグ（B* タグ）
- 計画中の改善（E* タグ）

### 3. 実際のコードを確認

ROADMAP と SPEC を照合しながら、主要ファイルを確認する:

**エンジン層** (`src/engine/`):
- `simulation.js` — シミュレーションロジックが SPEC の仕様を満たしているか
- `player.js` — 選手老化・引退が実装されているか
- `sabermetrics.js` — セイバーメトリクスが正しく計算されているか
- `awards.js` — 表彰・記録システムが機能しているか

**UI 層** (`src/components/`):
- `src/components/tabs/` — 各タブの機能が SPEC 通りか
- `TacticalGame.jsx` — 試合操作の UI が完成しているか
- `DashboardTab.jsx` — ダッシュボードに必要な情報が揃っているか

**状態管理**:
- `src/App.jsx` — game flow が SPEC の設計に沿っているか

**定数**:
- `src/constants.js` — バランス数値が SPEC の設計意図に沿っているか

### 4. NPB 協約適合チェック

以下のチェック項目について、`src/constants.js`・`src/engine/contract.js`・`src/hooks/useOffseason.js`・`src/engine/scheduleGen.js` を確認し、実際の NPB 協約への準拠状況を評価する:

| チェック項目 | 根拠 | 確認先ファイル |
|---|---|---|
| FA 資格日数（高卒8年/大卒7年/海外9年） | NPB 協約 第173条 | `src/engine/contract.js` |
| 最低年俸（支配下 420万/育成 240万） | NPB 協約 第89条 | `src/constants.js` |
| 外国人枠（一軍最大 4 名） | NPB 協約 第82条 | `src/constants.js`, `src/hooks/useGameState.js` |
| 外国人選手の国内選手扱い移行（960日） | NPB 協約 第82条 | `src/engine/contract.js` |
| 支配下登録上限（70名） | NPB 協約 第67条 | `src/constants.js` |
| 一軍登録人数（28名） | NPB 協約 第72条 | `src/constants.js` |
| 育成契約上限（3年） | NPB 協約 第87条 | `src/hooks/useOffseason.js` |
| シーズン試合数（143試合）・交流戦（18試合） | NPB 公式規程 | `src/engine/scheduleGen.js` |
| ポスティングシステム | MLB-NPB ポスティング協定 | ROADMAP.md（Tier 10） |
| 戦力外通告・ウェーバー制度 | NPB 協約 第100条前後 | ROADMAP.md（Tier 10） |

各項目を ✅（実装済み・準拠）/ ⚠️（部分実装・要確認）/ ❌（未実装・非準拠）で評価する。

### 5. SPEC.md 自体の評価

SPEC.md の設計が「理想の NPB シミュレーションゲーム」として妥当かを批判的に評価する。評価軸は以下の3点:

| 評価軸 | 問い |
|---|---|
| NPB らしさの欠如 | SPEC に書かれているが NPB の実態・慣習と乖離している仕様はないか（例: 実際にはありえない数値設計、NPB 独自の文化が反映されていない箇所） |
| 記述の陳腐化 | 実装が進んだ結果、SPEC の記述が実態と乖離していたり古くなっている箇所はないか |
| 設計ギャップ | SPEC に書かれていないが「NPB ゲームとして面白くするために必要な要素」は何か（例: 監督・コーチシステム、球団経営の財務圧力、選手の精神状態） |

### 7. ユーザーへの報告

レポートを提示し、「次の一手」のスキル（`/game-feature`, `/game-bugfix`, `/game-balance`）を案内する。

## 注意事項

- コードを修正しない。あくまで評価・提案に徹する
- 「理想のゲーム」の第一基準は「NPB シミュレーションとして面白く・正確であること」。SPEC.md はその写しであり、SPEC 自体が誤っている可能性も考慮して評価する
- NPB 協約との乖離は SPEC.md の記載有無に関わらず指摘する（SPEC 未記載でも NPB ルール上重要な項目は未実装リスクとして報告する）
- SPEC.md の設計ミス・陳腐化・設計ギャップを発見した場合は、実装との乖離と同等の優先度で報告する
