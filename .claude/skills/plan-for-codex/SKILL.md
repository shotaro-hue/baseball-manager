# plan-for-codex スキル

Claude のコンテキストウィンドウを節約しながら、Codex が自律実装できる高品質なタスク仕様書を生成するワークフロー。
**このスキルは実装しない。** 出力は `TASKS/<feature-name>.md` のみ。

## ワークフロー

Make a todo list for all the tasks in this workflow and work on them one after another.

### 1. ROADMAP を読んで候補を収集

`ROADMAP.md` を読み込み、以下の優先順で未実装タスクを最大3件収集する:

1. 現在進行中の Tier（🔶 部分実装）の中で未着手のタスク
2. 次の Tier（🔴 未着手）の先頭タスク
3. 既知のバグ（B* タグ）で未修正のもの

以下の形式で候補を提示し、**ユーザーの選択を待つ**:

```
## タスク候補

| # | タイプ | タスク | Tier | 理由 |
|---|---|---|---|---|
| 1 | feature / bugfix / balance / ui / refactor | タスク名 | X | 選定理由 |
| 2 | ... | ... | ... | ... |
| 3 | ... | ... | ... | ... |

どれを仕様書にしますか？（番号 / タスク名を直接指定 / キャンセル）
```

| ユーザーの反応 | 次のアクション |
|---|---|
| 番号または名前 | ステップ2へ |
| キャンセル | 終了 |

---

### 2. SPEC.md の該当セクションを精読

`SPEC.md` から、選択した機能に関係するセクションのみを読む（全体を読まない）。
把握すべき内容:
- 機能の目的・ゲームへの影響
- データモデルへの影響（新規フィールド等）
- UI 要件（あれば）
- NPB 協約上の制約（あれば）

---

### 3. ファイル・関数の場所を特定（軽量調査）

**目的**: Codex が「どこを読めばいいか」を迷わない程度の情報を収集する。
**制約**: ファイル全体は読まない。`Grep` / `Glob` と部分 `Read`（50〜100行程度）にとどめる。

確認すること:
- 機能に関係する `src/engine/` のファイル名と主要関数名（Grep で特定）
- 挿入ポイントの行番号のヒント（Grep で特定）
- `src/constants.js` に追加が必要な定数ブロックの場所
- 類似 UI コンポーネントのファイル名（新規 UI が必要な場合のみ）

この調査で得た情報を「読むべきファイル」テーブルと「実装ガイダンス」に反映させる。

---

### 4. 要件サマリーをユーザーに提示（承認ゲート）

以下の表形式で要件をまとめてユーザーに提示し、**承認を得てからステップ5へ進む**:

| 項目 | 内容 |
|---|---|
| 機能の目的 | |
| 入出力・state 影響 | |
| 受け入れ条件（主要3件） | |
| 制約・エッジケース | |
| 他機能との依存関係 | |
| SPEC 未定義・曖昧な点 | |
| NPB 協約上の制約 | なし / または条文番号と内容 |

| ユーザーの反応 | 次のアクション |
|---|---|
| **承認**（「OK」「進めて」等） | ステップ5へ |
| **修正依頼** | サマリーを修正して再提示 |
| **中断** | TASK ファイルを生成せず終了 |

---

### 5. TASKS/<feature-name>.md を生成

`TASKS/` ディレクトリに以下のフォーマットでファイルを書き出す。
ファイル名は kebab-case（例: `TASKS/farm-simulation.md`）。

```markdown
---
task-id: <kebab-case-identifier>
type: feature | bugfix | balance | ui | refactor
commit-prefix: feat | fix | balance | refactor
created: YYYY-MM-DD
roadmap-item: "㉔ 機能名 (Tier X)"
---

# Task: <機能名>

## 背景・目的
（2〜4文。なぜ必要か・ゲームへの影響。Codex がコンテキストゼロで読んでわかる精度で書く）

## 機能説明
（実装すべき動作を具体的に記述。「〜のとき〜する」形式）

## 読むべきファイル（優先順）
| ファイル | 理由・参照ポイント |
|---|---|
| `src/engine/foo.js` | `barFn()` を拡張する（~120行目付近） |
| `src/hooks/useSeasonFlow.js` | 週次処理の挿入ポイント（~180〜220行） |
| `src/constants.js` | 定数ブロックの追加場所を確認 |

## 変更するファイル
| ファイル | 操作 | 備考 |
|---|---|---|
| `src/engine/foo.js` | Modify | `newFn()` を追加 |
| `src/constants.js` | Modify | 新定数ブロックを末尾に追加 |
| `src/components/tabs/XxxTab.jsx` | Create / Modify | 説明 |

## 実装ガイダンス

### Step 1: 定数追加（src/constants.js）
（具体的な内容。変数名・型・意図まで書く）

### Step 2: エンジン実装
（関数シグネチャ・処理の流れ・利用すべき既存関数を記述）

### Step 3: hooks 更新
（state の変更場所・呼び出しタイミングを記述）

### Step 4: UI（該当する場合のみ）
（コンポーネント構成・参照すべき既存コンポーネントを記述）

## データモデル変更
（新規フィールドがある場合のみ）
```js
// src/engine/player.js の generatePlayer() に追加
newField: [],  // 何を記録するか
```

## 受け入れ条件
- [ ] X のとき Y になる
- [ ] Z の場合は処理されない
- [ ] ビルド・全テスト通過

## テストケース
- `src/engine/__tests__/foo.test.js` に `describe("newFn")` を追加
  - 正常系: input A → output B
  - エッジケース: null → デフォルト値を返す

## NPB 協約上の制約
（関係なければ「なし」）

## 過去バグからの教訓
（CLAUDE.md の教訓テーブルから関連項目をコピー）
- B1 パターン: 両チームに適用すること（該当する場合）

## コーディング規約リマインダー
- `Math.random()` 禁止。`rng()` / `rngf()` を使う（`src/utils.js`）
- ゲームバランス数値は `src/constants.js` に定数として切り出す
- 選手・イベント ID は `uid()` で生成（`src/utils.js`）

## ROADMAP.md 更新指示
- `<タスク番号>` の状態を `🔴` → `✅` に変更
- 「最終更新」ヘッダー行を `YYYY-MM-DD（<機能名> 完了）` に更新

## CHANGELOG.md エントリ（コミット後に hash を埋めること）
```
### YYYY-MM-DD — <機能名>（コミットハッシュ）

**仕様本文への影響あり（§X・§Y）** or **仕様本文への影響なし（内部実装のみ）**

- 実装内容の箇条書き
```

## SPEC.md 更新箇所
（変更がある場合のみ。なければ「なし」）
- §4.X — [具体的な変更内容]
- §5 データモデル — [新フィールドの追記]

## コミットメッセージ
`feat: <簡潔な説明（日本語可）>`

## PR タイトル
`feat: <コミットメッセージと同じ>`
```

---

### 6. 完了報告

以下の形式で報告する:

```
## plan-for-codex 完了

- 生成ファイル: TASKS/<name>.md
- 対象機能: <機能名>（ROADMAP <タスク番号>）
- 事前調査済みファイル: <一覧>

## 次のステップ（Codex セッションで実行）

/implement-from-task TASKS/<name>.md
```

## 注意事項

- **コードを書かない・ファイルを変更しない**（`TASKS/` への Write のみ許可）
- ステップ3の調査は「場所の特定」に留める。実装詳細の把握は Codex の仕事
- SPEC.md の記述が曖昧な場合は、ステップ4の承認ゲートでユーザーに方針を確認してから TASK ファイルに明記する
- NPB 協約との整合に懸念があれば必ず TASK ファイルの「NPB 協約上の制約」に記録する
