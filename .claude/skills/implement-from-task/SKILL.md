# implement-from-task スキル

`TASKS/<name>.md` に記載された仕様を読み、承認ゲートなしで実装を完走するワークフロー。
Claude セッションでの承認は `/plan-for-codex` 実行時に済んでいるため、このスキル内では確認不要。

## 呼び出し方

```
/implement-from-task TASKS/<feature-name>.md
```

引数なしで呼んだ場合は `TASKS/` ディレクトリの `.md` ファイル一覧を表示し、
ユーザーに選択を求める（1回のみ）。

## ワークフロー

Make a todo list for all the tasks in this workflow and work on them one after another.

### 1. TASK ファイルを読み込む

指定された `TASKS/<name>.md` を Read する。
以下の情報を必ず把握してから次のステップへ進む:
- `task-id`, `type`, `commit-prefix`
- 「読むべきファイル」テーブル（全件）
- 「変更するファイル」テーブル（全件）
- 「実装ガイダンス」の全ステップ
- 「受け入れ条件」チェックリスト

### 2. 参照ファイルをすべて読む

TASK ファイルの「読むべきファイル」に列挙されたファイルをすべて Read する。
**列挙されていないファイルは読まない**（Grep で補足調査は可）。
CLAUDE.md も Read してコーディング規約を確認する。

### 3. 実装

「実装ガイダンス」の Step を順番に実行する。

守るべき規約（TASK ファイルの「コーディング規約リマインダー」にも記載）:
- `Math.random()` 禁止 → `rng()` / `rngf()`（`src/utils.js`）を使う
- ゲームバランス数値 → `src/constants.js` に名前付き定数として切り出す
- 選手・イベント ID → `uid()`（`src/utils.js`）で生成
- ゲームロジックを UI 層に書かない

「変更するファイル」テーブルが変更範囲の正規リスト。テーブル外のファイルを変更する場合は
その理由を完了報告に記載する。

### 4. ビルド・テスト確認

`.claude/skills/_shared/build-and-test.md` を Read して記載の手順を実行する。

ビルド・テストが失敗した場合:
1. エラーを読んで原因を特定する
2. 自律的に修正を試みる（最大3回）
3. 3回修正しても解消しない場合のみ、エラー内容をユーザーに報告して停止する

### 5. ROADMAP.md を更新

TASK ファイルの「ROADMAP.md 更新指示」に従って更新する。
指示通りに実行するだけ。推測・拡大解釈しない。

### 6. CHANGELOG.md + SPEC.md を更新

**CHANGELOG.md**:
TASK ファイルの「CHANGELOG.md エントリ」テンプレートをそのまま使い、
`CHANGELOG.md` の先頭（最新エントリより上）に追記する。
コミットハッシュはこの時点では `（実装完了）` と仮記入し、ステップ7後に実際のハッシュで更新する。

**SPEC.md**:
TASK ファイルの「SPEC.md 更新箇所」に「なし」と記載されている場合は更新しない。
更新箇所が指定されている場合のみ、指示された箇所を更新する。

### 7. コミット・プッシュ

`.claude/skills/_shared/commit-push-pr.md` を Read して記載の手順を実行する。

**PR 承認はスキップする**（`plan-for-codex` セッションで承認済みのため）。
PR 作成・マージは自動で実行する。

コミットプレフィックスと本文は TASK ファイルの「コミットメッセージ」セクションの通りに使う。

コミット完了後、実際のコミットハッシュで CHANGELOG.md の仮記入部分を更新する。

### 8. 完了報告

以下の形式で報告する:

```
## implement-from-task 完了

- 使用した TASK ファイル: TASKS/<name>.md
- 実装した機能: <機能名>
- 変更したファイル: <一覧>
- ビルド・テスト: ✅ 成功 / ‼️ 失敗（詳細）
- マージ: PR #<番号>

## 受け入れ条件チェック
- [x] X のとき Y になる
- [x] ビルド・全テスト通過
- [ ] （未達成の項目があれば理由も記載）

## 次のステップ
Claude セッションで次のタスクを計画する場合: /plan-for-codex
```

## 注意事項

- TASK ファイルが仕様書。それ以外の解釈・拡大実装をしない
- ユーザーへの確認は「ビルド3回失敗」以外は行わない
- TASK ファイルに記載のない機能をついでに実装しない
- 「変更するファイル」テーブル外のファイルを変更した場合は完了報告に明記する
